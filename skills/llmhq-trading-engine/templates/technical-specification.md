# LLM Heuristic Quant (LLMHQ)
## Second Refinement Document
### OpenClaw-Oriented Technical Specification for a Custom Skill / Project Runbook

## 1. Document Purpose

This document is a **second refinement** of the LLMHQ concept. It merges the stronger structure of the first refinement with important details from the earlier draft that should not be lost. The result is intended to serve as the **source specification** for creating a production-quality `SKILL.md` and related project files for an OpenClaw implementation.

This is **not** the final `SKILL.md`. Instead, it is the authoritative design document that:

- consolidates the original concept and the first rewrite,
- restores useful technical and strategic details that were compressed or omitted,
- clarifies operating assumptions,
- separates architecture, timing, tooling, and retrospective learning logic,
- makes the project easier to convert into an OpenClaw skill or a small skill bundle.

---

## 2. Executive Summary

LLMHQ is a **quantitative trading decision-support and execution orchestration system** for ultra-short-horizon BTC interval prediction, initially focused on **Polymarket 5-minute and 15-minute BTC resolution windows**.

The central design principle is that the system should not rely on a large language model to process raw market noise directly. Instead, the LLM acts as a **Chief Investment Officer (CIO)** or **strategy layer** that consumes precomputed, semantically meaningful summaries generated by deterministic utilities.

In practical terms:

- **Rust** handles fast data collection, feature calculation, timing-sensitive orchestration, and execution.
- **The LLM** handles context synthesis, regime interpretation, veto logic, and adaptive weighting of signals.
- **A retrospective loop** logs outcomes, attributes wins and failures, and updates the decision logic over time.

The intended result is a hybrid system that combines:

- low-latency execution,
- richer contextual reasoning,
- post-trade learning,
- and a structured path to deployment within an **OpenClaw skill framework**.

## 3. Core Design Philosophy

### 3.1 From “coding for conditions” to “coding for context”

Traditional trading bots are often built as rigid condition engines:

- if RSI < X, buy,
- if MACD crosses Y, sell,
- if spread widens, reduce risk.

LLMHQ deliberately moves away from this brittle pattern. The deterministic layer still computes indicators and microstructure features, but the final decision is based on **contextual interpretation**, not isolated thresholds.

The LLM should answer questions such as:

- Is this move genuine or a fake-out?
- Is order flow aligned with momentum or contradicting it?
- Is the market trending, compressing, pinning, or mean-reverting?
- Which signals deserve the most weight in the current regime?

This allows the system to behave more like a human quant strategist reviewing analyst notes rather than a single-rule trading bot.

### 3.2 Separate calculation from interpretation

The LLM should **not** calculate raw indicators from scratch during the critical trading window. Deterministic utilities should produce the numerical facts first. The LLM should then interpret those facts.

This separation is essential for:

- latency,
- repeatability,
- audit-ability,
- and cleaner post-mortem analysis.

### 3.3 Prefer semantic abstractions over raw noise

A key concept preserved from the early draft is that the LLM performs better when it receives **high-fidelity abstracts**, not unstructured tick or order book streams.

Examples:

- instead of raw depth tables, provide: “sell-side walls thinning near the ask”;
- instead of raw HMA values, provide: “price is surfing the 14-period HMA with steepening slope”;
- instead of raw OBI values, provide: “buy-side absorption is persistent and the sell book is shallow.”

This translation layer is one of the most important features in the entire design.

## 4. Primary Objective

The primary objective is to predict whether the next **5-minute** or **15-minute** BTC interval will resolve **UP** or **DOWN**, with enough confidence to justify a trade placed at the opening of the new interval (t=0).

The specific execution concept is a **blind trade at `t=0`**:

- the new block has not yet materially unfolded,
- the system is trading the residual kinetic energy, structure, and microstructure setup of the immediately preceding block,
- the decision must therefore be made **before** the new block meaningfully develops.

This makes timing and pre-computation critical.

### 5.1 In scope

- BTC prediction for Polymarket 5-minute and 15-minute markets,
- pre-trade market state synthesis,

## 6. System Roles and Mental Model

The earlier draft introduced a strong metaphor that should be retained because it makes the design much clearer.

### 6.1 The LLM as the “CIO”

The LLM is not the calculator. It is the strategic decision-maker.

Responsibilities:

- review the latest market narrative,
- assess alignment or conflict between signals,
- infer regime,
- apply veto logic,
- produce final direction and confidence,
- contribute to retrospective learning.

### 6.2 Deterministic utilities as the “Analysts”

Specialised utilities generate structured observations that feed the CIO.

These analysts include both the explicitly named components from the draft and the more advanced microstructure modules introduced in the refinement.

Core analyst roles:

- **Tape Reader** – interprets L2 depth and order book pressure,
- **Momentum Engine** – computes short-horizon momentum features and divergence clues,
- **Whale Watcher** – detects large exchange inflows, liquidations, or other high-impact flow,
- **Social Sentiment Filter** – extracts high-signal market sentiment from selected sources,
- **Microstructure Engine** – computes order-flow toxicity, imbalance velocity, voids, and other short-horizon features,
- **Cross-Exchange Lead-Lag Monitor** – checks whether perp, spot, or related venues are leading or lagging.

### 6.3 The retrospective layer as the “Coach”

The Coach reviews the system’s decisions after the fact.

Responsibilities:

- compare forecast with outcome,
- determine which signals were useful or misleading,
- identify regime-specific failures,
- generate lessons learned,
- propose updated weighting or prompt logic.

This role is critical and should remain explicit in the final project design.

## 7. High-Level Architecture

The architecture can be described as an **Observer → Synthesiser → CIO → Execution → Retrospective** pipeline.

### 7.1 Layer A: Real-Time Sensory Array

This is the data intake and feature generation layer.

Inputs may include:

- exchange OHLCV,
- spot and perp price feeds,
- L2 order book data,
- order book imbalance,
- liquidation feeds,
- whale transfer feeds,
- curated sentiment sources,
- cross-exchange spread data.

Outputs from this layer should be:

- raw snapshots for logging,
- derived indicators for deterministic review,
- semantically translated narratives for LLM consumption.

### 7.2 Layer B: Semantic Synthesis

This layer converts numeric and structural observations into concise, high-context language that a strategy model can evaluate quickly.

This can be produced by:

- deterministic template logic,
- a smaller/faster LLM,
- or a hybrid of both.

The refinement document’s **Semantic Analyst** concept is useful and should be kept. It reduces cognitive load on the primary strategy model.

### 7.3 Layer C: Decision Core

The primary LLM, acting as the **PhD CIO**, receives:

- raw snapshot,
- derived indicators,
- semantic narrative,
- optional known external context,
- current regime assumptions,
- policy instructions and veto rules.

It then outputs:

- `UP` or `DOWN`,
- confidence score,
- brief rationale,
- lead driver,
- risk flags,
- any veto or caveat.

### 7.4 Layer D: Execution Layer

The Rust execution layer:

- prepares the transaction,
- enforces timing discipline,
- applies minimum confidence thresholds,
- optionally applies last-moment safety checks,
- submits the trade at or immediately before the interval boundary.

### 7.5 Layer E: Retrospective Loop

After the trade resolves sufficiently:

- outcome is recorded,
- the system performs a ghost-trade analysis,
- daily meta-review updates operating guidance,
- any new prompt or policy changes should be versioned and auditable.

## 8. Why the System Must Use a Hybrid Rust + LLM Approach

The original draft correctly framed this as solving the **latency versus logic** dilemma.

### 8.1 Why Rust

Rust is the correct execution-layer choice for:

- WebSocket streaming,
- concurrent feature calculation,
- precise timing,
- deterministic pipelines,
- low overhead,
- strong safety guarantees.

Recommended core implementation areas for Rust:

- feed collection,
- indicator calculations,
- buffer management,
- transaction orchestration,
- database logging,
- outcome reconciliation.

### 8.2 Why an LLM is still necessary

Purely deterministic systems can miss contextual patterns such as:

- contradictory evidence across timeframes,
- fake-outs,
- regime shifts,
- sentiment-versus-flow conflicts,
- manipulation patterns that do not map cleanly to fixed rules.

The LLM provides the missing contextual layer, but only if it is fed structured evidence instead of raw noise.

## 9. Time Horizon and Blind-Trade Logic

This section should remain prominent because it is central to the strategy.

At `t=0`, the system is not reacting to the new candle’s development. It is betting on the **stored kinetic energy** or **structural setup** at the close of the prior interval.

The system is therefore trying to determine whether the just-finished block implies:

- **continuation**,
- **immediate mean reversion**,
- **liquidity hunt then reversal**,
- or **compression followed by expansion**.

That means the decision logic should heavily evaluate:

- block close conditions,
- last 30 seconds of order book behaviour,
- acceleration or decay of momentum,
- imbalance velocity,
- location of liquidity voids,
- cross-venue pressure,
- known whale or liquidation events,
- potential pinning or spoofing.

## 10. Decision Timing Model

The timing model from the refinement is good and should be preserved, with clearer wording.

### 10.1 Recommended timeline

#### `t-30s` to `t-15s`: parallel feature calculation

Rust calculators run in parallel to compute:

- order book imbalance,
- OBI velocity,
- HMA and slope,
- VPIN or equivalent toxicity measures,
- liquidity void mapping,
- price velocity and deceleration,
- spread delta across venues,
- any sentiment or whale alerts already available.

#### `t-15s` to `t-10s`: aggregation

The orchestrator combines:

- raw market snapshot,
- derived features,
- regime clues,
- risk annotations.

#### `t-10s` to `t-5s`: semantic synthesis

A semantic layer converts the payload into a brief “story of the next block.”

#### `t-5s` to `t-2s`: CIO decision window

The primary strategy LLM reviews:

- raw facts,
- derived indicators,
- semantic interpretation,
- policy constraints,
- veto rules.

It produces:

- direction,
- confidence,
- supporting rationale,
- key driver,
- caution flags.

#### `t-2s` to `t=0`: execution preparation

The Rust execution layer:

- validates the decision,
- confirms confidence threshold,
- stages the transaction,
- fires at the exact boundary (t=0).

### 10.2 Why this timing model matters

It shifts heavy cognitive work earlier in the cycle so the final decision is a review of a structured thesis rather than a cold inference from scratch.

## 11. Market Regime Logic

One important idea from the early draft that should be explicitly restored is **regime-based thresholding**.

The strategy should not use the same weighting in every market state.

### 11.1 Core regimes

The first version should classify at least these regimes:

- **Trending**
- **Ranging / Mean Reverting**
- **Volatile Expansion**
- **Quiet Compression**
- **Manipulative / Pinning / Spoof-prone**

### 11.2 Regime-specific interpretation examples

- In a **trend** regime, momentum and microstructure continuation matter more than overbought oscillators.
- In a **mean-reversion** regime, RSI/Bollinger-style exhaustion signals regain value.
- In a **breakout/compression** regime, liquidity voids and acceleration are more important than sentiment.
- In a **panic** regime, social sentiment and whale deposits can override otherwise bullish structure.
- In a **manipulative end-of-block** regime, extreme final-seconds OBI velocity may be treated as a warning, not confirmation.

This adaptive weighting is a foundational concept and should not be lost in any future conversion into `SKILL.md` or system prompts.

## 12. Analyst Utilities and Signals

This section consolidates the best ideas from both drafts.

### 12.1 Core analyst utilities

#### Tape Reader
Purpose:
- convert order book depth and L2 dynamics into actionable pressure summaries.

Typical outputs:
- buy-wall persistence,
- sell-wall thinning,
- absorption versus retreat,
- path of least resistance.

#### Momentum Engine
Purpose:
- summarise short-horizon momentum structure.

Potential calculations:
- ROC,
- RSI,
- MACD,
- HMA slope,
- divergence,
- price velocity.

Typical outputs:
- “bearish divergence forming on the 1-minute timeframe,”
- “upward momentum is decelerating into the close,”
- “trend slope is steepening without visible exhaustion.”

#### Whale Watcher
Purpose:
- detect large transfers, exchange inflows, or liquidation events likely to affect near-term direction.

Typical outputs:
- large BTC inflow to exchange,
- liquidation cluster near current price,
- suspicious high-impact transfer activity.

#### Social Sentiment Filter
Purpose:
- monitor curated, high-signal channels for emotionally important events or sudden sentiment shocks.

This should remain tightly scoped and not become a noisy full social-firehose feature.

Typical outputs:
- slight fear,
- pump narrative emerging,
- panic spike tied to macro headline.

### 12.2 Advanced microstructure utilities

#### OBI Pressure Gauge
Measures order book imbalance and side dominance.

#### OBI Velocity
Measures how fast imbalance changes in the last seconds before block close.

#### HMA Momentum Pulse
Summarises Hull Moving Average direction and slope.

#### Jitter Filter
Separates genuine compression or structure from meaningless noise.

#### Velocity Vector
Captures whether price movement is accelerating, decelerating, grinding, or spiking.

#### Liquidity Map / Liquidity Void Detector
Identifies likely stop clusters, thin zones, and air pockets above or below price.

#### VPIN / Order Flow Toxicity
Distinguishes informed flow from benign flow and estimates whether market makers are being run over.

#### Cross-Exchange Lead-Lag Monitor
Tracks whether spot, perp, Deribit, BitMEX, Binance, Coinbase, or other selected venues are leading the move.

### 12.3 Preserved concepts from the original draft

Several concepts were useful but became less explicit in the refinement. They should be retained:

- **Liquidity_Map()** – identify likely stop-loss clusters,
- **Velocity_Delta()** – distinguish slow exhaustion from fast continuation,
- **Correlation_Checker()** – detect whether BTC is being confirmed or contradicted by ETH/SOL or other related market leaders.

These do not need to keep exactly those function names, but the underlying logic should remain in scope.

## 13. Semantic Translation Layer

This layer is so important that it deserves its own section.

The translation layer should map raw numbers into compact semantic summaries. This can be a fast LLM.

### 13.1 Example transformations

Instead of:
- `HMA14 slope = +12°`

Use:
- “Price is surfing the 14-period HMA; slope is steepening and no visible exhaustion is present.”

Instead of:
- `OBI = 0.80`

Use:
- “Buy-side absorption is dominant and the sell-side depth is unusually thin.”

Instead of:
- `VPIN = 0.82`

Use:
- “Market toxicity is elevated; liquidity providers appear to be pulling back under informed pressure.”

### 13.2 Requirements for this layer

The semantic output must be:

- concise,
- consistent,
- auditable,
- suitable for storage in logs,
- comparable across trades,
- stable enough to support retrospective analysis.

It should not produce poetic or overly long prose. It is an operational briefing, not a narrative essay.

## 14. CIO Decision Logic

The CIO must not simply parrot the semantic layer. It must verify, challenge, and possibly veto the story.

### 14.1 Required decision responsibilities

The CIO should:

1. verify whether raw data supports the semantic story,
2. identify contradictions,
3. assess regime,
4. decide whether the move is continuation or reversion,
5. apply veto logic where necessary,
6. issue a final binary direction with confidence.

### 14.2 Veto logic examples

Examples preserved from the draft:

- microstructure says **UP**, but whale activity indicates a likely dump,
- momentum looks bullish, but social panic or macro shock argues against continuation,
- a final-seconds imbalance spike may actually indicate pinning or spoofing rather than genuine demand.

### 14.3 Minimum CIO output format

Each decision should produce at least:

- `direction`: `UP` or `DOWN`
- `confidence`: integer 0–100
- `regime`: one of the active market states
- `lead_driver`: e.g. OBI, VPIN, HMA, sentiment, whale flow
- `rationale`: short explanation
- `risk_flags`: array of warnings
- `veto_applied`: yes/no and why

This structure will make downstream logging and post-mortems much easier.

## 15. Data Payload to the CIO

The refined draft’s “Briefing Dossier” concept is strong and should be kept.

At decision time, the CIO should receive three layers.

### 15.1 Layer 1: Raw data snapshot

Examples:
- price,
- best bid/ask,
- depth within selected range,
- perp premium,
- spread deltas,
- recent liquidation markers.

### 15.2 Layer 2: Derived indicators

Examples:
- OBI,
- OBI velocity,
- HMA slope,
- VPIN,
- volatility state,
- liquidity void presence,
- price acceleration or deceleration.

### 15.3 Layer 3: Semantic narrative

A concise paragraph or bullet briefing describing the likely structure of the next block.

This three-layer approach is cleaner than sending only raw data or only prose.

## 16. Execution Logic

The execution layer must be deterministic and conservative.

### 16.1 Execution requirements

The Rust layer should:

- only execute if confidence exceeds threshold,
- confirm decision arrived in time,
- reject stale decisions,
- log final pre-trade state,
- include latency measurement,
- support dry-run mode for testing.

### 16.2 Suggested safety checks

Before placing a live trade, the system should verify:

- market boundary timing is correct,
- exchange or market feed is healthy,
- decision confidence is above minimum,
- any hard veto flag is absent,
- transaction path is available,
- database logging is available or durable fallback exists.

### 16.3 Blind trade still needs guardrails

Although the strategy is intentionally early and aggressive, the system should never place a trade merely because a direction exists. Confidence and safety gates should remain mandatory.

## 17. Logging, Memory, and Post-Mortem Design

This is another critical section and should remain detailed.

### 17.1 Database choice

A **Postgres/TimescaleDB** design is appropriate for:

- structured trade logs,
- interval-based review,
- feature attribution,
- trend analysis over many blocks.

### 17.2 Minimum trade log schema

Each logged decision should store at least:

- timestamp,
- interval type (5m or 15m),
- market regime,
- raw snapshot hash or reference,
- semantic narrative,
- prediction,
- confidence,
- lead driver,
- veto notes,
- outcome,
- execution latency,
- slippage or execution timing data,
- retrospective lesson if generated.

### 17.3 Why logging the narrative matters

The original drafts correctly recognised that logging only numbers is not enough. The **exact semantic narrative sent to the LLM** is a first-class artefact and must be stored so that later reviews can determine whether the story itself was wrong, incomplete, or contradicted by the raw data.

## 18. Ghost-Trade Analysis

The refinement draft introduced the “ghost-trade” concept well. It should remain as a named subsystem.

### 18.1 Purpose

Shortly after trade entry, the system should analyse the first segment of the new block to answer:

- did the market initially confirm the thesis?
- did it reverse immediately?
- was there a fake wall, spoof, whale event, or last-second manipulation?

### 18.2 Example investigative questions

- Was the OBI signal real or a fake wall?
- Did a whale or liquidation event appear only after the decision?
- Was the HMA lagging during a volatility expansion?
- Did end-of-block pinning create a false continuation signal?

### 18.3 Outputs

Ghost-trade analysis should produce:

- classification of failure or success,
- likely cause,
- lesson learned,
- suggestion for regime-specific weight adjustment,
- optionally a warning rule for future similar conditions.

## 19. Self-Tuning and RLAIF

The self-tuning loop from both drafts should be preserved, but with clearer constraints.

### 19.1 What it should do

At a regular cadence, likely daily for the first version, the system should review recent trades and identify:

- weak indicators,
- false positives,
- regime-specific failure patterns,
- over-weighted and under-weighted drivers,
- prompt or policy improvements.

### 19.2 What it should not do automatically in v1

The first live version should **not** allow unconstrained autonomous prompt mutation directly into production without review. Instead, it should:

- generate a proposed policy update,
- store it as a versioned artefact,
- allow review or controlled activation.

This preserves the valuable RLAIF concept while avoiding silent drift.

### 19.3 Example review output

- “HMA-led long signals underperform during volatile closes unless OBI exceeds 0.6.”
- “Late OBI spikes in the final 5 seconds often represent pinning and should reduce confidence.”
- “Sentiment vetoes are too strong in ranging markets and should be softened.”

## 21. OpenClaw Translation Notes

Because the goal is to convert this into an OpenClaw project and eventually a `SKILL.md`, the design should be expressed in a way that maps cleanly into the current OpenClaw skill model.

A `SKILL.md` is a directory-contained markdown file with YAML frontmatter and step-by-step instructions, and OpenClaw loads eligible skills based on frontmatter and environment gating. Skills can be installed from bundled locations, `~/.openclaw/skills`, or workspace `skills/` directories. `SKILL.md` should at minimum define `name` and `description`, and optional fields such as `user-invocable`, `disable-model-invocation`, and single-line JSON `metadata` can control invocation and gating. citeturn287727view0turn303807view0

### 21.1 Recommended OpenClaw approach

This project will likely be better represented as either:

- one primary skill with clearly defined subcommands and external scripts, or
- a small skill bundle split by responsibility:
  - data snapshot / analysis,
  - prediction run,
  - post-mortem review,
  - policy review.

Given the tutorial guidance to keep skills narrow and to use `disable-model-invocation` for broader or riskier workflows, it may be wise to keep live execution commands manual or tightly gated at first. citeturn287727view0turn303807view0

### 21.2 Recommended `SKILL.md` responsibilities

The future skill should clearly describe:

- what the skill does,
- when it should be invoked,
- what binaries, env vars, and config it requires,
- which commands are read-only versus execution-capable,
- explicit stop conditions,
- expected output format,
- logging requirements,
- safety restrictions.

### 21.3 Likely frontmatter needs

The eventual `SKILL.md` will probably need:

- `name`
- `description`
- `disable-model-invocation: true` for risky execution functions in early versions
- `user-invocable: true` for manual runs
- `metadata.openclaw.requires.bins`
- `metadata.openclaw.requires.env`
- possibly `metadata.openclaw.requires.config`

OpenClaw’s docs also note that `metadata` must remain a **single-line JSON object**, and gating can depend on required binaries, environment variables, OS, and config values. citeturn303807view0

### 21.4 Suggested skill safety stance

The LumaDock tutorial emphasizes that skills are operational instructions, not isolated code sandboxes, and warns against broad or risky skills without clear guardrails. It also recommends explicit confirmation for destructive actions and use of `disable-model-invocation` where appropriate. citeturn287727view0

For LLMHQ, that implies:

- read-only market analysis commands can be model-invocable later,
- live trade execution should initially be manual or separately gated,
- secrets should come from env/config, not prompt text,
- output should be deterministic and structured.

To make the consolidation explicit, here are the main elements that needed to be restored or clarified from the earlier draft:

1. **Analyst role framing** – Tape Reader, Momentum Engine, Whale Watcher, and Social Sentiment Filter remained useful and should stay visible.
2. **Regime-based thresholding** – this is strategically important and should not be buried.

6. **Safer self-tuning framing** – retain RLAIF, but under controlled policy versioning.

## 23. Recommended MVP Build Sequence

### Phase 1: Read-only analysis MVP

Build a version that:

- collects data,
- computes features,
- generates semantic narrative,
- produces prediction and confidence,
- logs everything,
- does not place live trades.

### Phase 2: Paper execution / dry run

Add:

- interval boundary logic,
- simulated `t=0` execution,
- latency measurement,
- post-mortem analysis,
- daily policy review.

### Phase 3: Controlled live execution

Only after sufficient evidence:

- enable live execution,
- enforce hard confidence thresholds,
- maintain manual approval or limited rollout,
- keep execution capability out of broad automatic invocation.

### Phase 4: OpenClaw packaging

Convert stable workflows into:

- `SKILL.md`,
- supporting scripts,
- config docs,
- test prompts,
- runbook examples.

## 25. Final Consolidated Position

LLMHQ should be implemented as a **hybrid quant-intelligence system** where deterministic Rust components do the fast sensing and calculation, while the LLM acts as a **strategic decision engine** that reasons over a structured market briefing.

The best ideas from both source documents are compatible and should be combined, not chosen between.

The final architecture should preserve all of the following:

- semantic abstraction over raw noise,
- analyst/CIO/coach separation,
- regime-aware weighting,
- microstructure-first decisioning,
- blind-trade timing discipline,
- structured post-mortems,
- controlled self-tuning,
- and OpenClaw-safe packaging with explicit gating and guardrails.

This document should now be treated as the **authoritative refinement source** for the next step: drafting the actual `SKILL.md`, command structure, and supporting files.
