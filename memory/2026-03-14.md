## PolyTrader OBI-10 Stuck at -23.7% Fix

### Problem
User reported OBI-10 value stuck at -23.7% (displayed as -0.24) since block 1773440700 (10:20pm UK time on March 13th). All subsequent blocks showed the same frozen value.

### Investigation
1. Checked database - gen2_metrics table was empty
2. Examined ws_tasks.rs - WebSocket order book updates were working (600 updates/min)
3. Checked logs - OBI avg consistently showing -0.24 with no variation
4. **Critical Finding**: TWO instances of poly-hft-pro were running simultaneously:
   - PID 489655 (old instance from Mar 13)
   - PID 521676 (current instance)

### Root Cause
Shared state corruption from duplicate process conflict. Both processes were competing for the same order book WebSocket connection and writing to shared obi_history buffer.

### Fix Applied
```bash
# Kill duplicate/old process
kill -9 489655

# Restart service via systemd
systemctl restart polytrader-5m
```

### Verification
- Service now running cleanly with single PID 541289
- WebSocket reconnected successfully
- Auto-restart enabled for persistence

### Prevention
Always verify only one instance is running:
```bash
ps aux | grep poly-hft-pro | grep -v grep
```

If multiple PIDs found, kill extras and restart service.

## Session 2: LLMHQ Data Flow Fix & PolyTrader Config

### LLMHQ Dashboard - Data Not Flowing
**Problem**: Dashboard showed static mock price ($82,543.21) instead of live BTC data.

**Root Cause**: 
- `CompleteLayerAOrchestrator.start()` set callback BEFORE `self.base.start()`
- But `LayerAOrchestrator.start()` overwrote it with its own `_on_candle` handler
- `connect()` runs infinite WebSocket loop, so post-start callback never executed

**Fix Applied**:
1. Modified `layer_a_collector.py` - Only set callbacks if `None`:
```python
if self.collector.on_candle is None:
    self.collector.on_candle = self._on_candle
```

2. Modified `layer_a_complete.py` - Set callback BEFORE start:
```python
self.base.collector.on_candle = self._on_candle_complete
await self.base.start()
```

3. Added `_on_trade_complete` for live price updates (every 5th trade):
```python
async def _on_trade_complete(self, tick):
    if len(self.base.microstructure.trade_history) % 5 == 0:
        self._save_data(tick.price)
```

**Result**: Price now updates every 2-3 seconds from Binance live feed.

## PolyQuant MA25 & MA7 Data Display Fix

### Problem
MA25 and MA7 values showing as 0 in UI despite 2215 rows of historical data in database.

### Root Cause 1: Insufficient Buffer
SMA calculator's `price_history` buffer only stored 20 minutes (`maxlen=1200`) but MA7 on 15M timeframe needs 105 minutes (6300 seconds) of data.

**Fix:**
```python
# Before
self.price_history = deque(maxlen=1200)  # 20 min

# After  
self.price_history = deque(maxlen=7200)  # 120 min
```

### Root Cause 2: Unrealistic Coverage Requirement
`calculate_sma()` required 80% data coverage - impossible with 5-second sampling.

**Fix:**
```python
# Before (required 5040 points for 15M MA7)
min_required = int(period_seconds * 0.8)

# After (realistic for 5s sampling)
expected_samples = period_seconds // 5
min_required = int(expected_samples * 0.5)
```

### Root Cause 3: Empty History Buffer on Startup
Decision model's internal buffer started empty, returning None for slope calculations.

**Fix:**
Changed `get_historical_ma()` to query database directly instead of relying on memory buffer:
```python
cursor.execute(f"""
    SELECT {ma_type}, timestamp 
    FROM second_ticks 
    WHERE timestamp BETWEEN ? AND ?
    ORDER BY ABS(timestamp - ?) ASC
    LIMIT 1
""")
```

### Result
- MA7_15m: Now showing actual values (e.g., $70,992.94)
- MA25_15m: Populating as data accumulates
- All timeframe MAs working correctly

|--------|-------|
| PIN Detected | ✅ YES |
| PIN Target | **YES** (defending UP) |
| PIN Risk Level | HIGH_HOLD |
| PIN Strength | 75% |
| Prior Block Slope | **+44.31°** (strong bullish) |
| Breakout Risk | **89.94%** |

### Classification Logic
- **HIGH_HOLD**: Pin supports regime direction (favorable)
- **HIGH_BREAK**: Pin fights regime direction (avoid)
- **LOW**: No pin detected

### Why Trade Still Got Hit
Despite HIGH_HOLD classification, 89.94% breakout risk meant the pin was stretched thin. When it broke, price moved against the held direction (YES).

### Integration
Added PIN detection as **FILTER 4** in ALT1 decision chain:
1. Regime Detection (15M MA7 slope)
2. Confirmation Slope Check (1M MA7 slope)
3. Opening Odds Divergence Check
4. **PIN Risk Assessment** ← Added back

**Features:**
- Detects HIGH_HOLD / HIGH_BREAK / LOW conditions
- Adjusts confirmation threshold to 5° for HIGH_HOLD (piggyback mode)
- Reduces position size when breakout risk elevated
- Blocks trades when pin risk unacceptable

- 15M MAs: MA7, MA25 (900-second periods)

**Decision Rules:**
1. **Regime Detection** (15M MA7 slope, t-300 to t-10):
