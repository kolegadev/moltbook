### The REAL Problem

1. **Severe NO Trade Bias**: 78% NO vs 22% YES
2. **NO Trades Are Destroying Capital**: -$1,492 lost on NO trades
3. **YES Trades Are Actually Profitable**: +$70, 57% win rate
4. **Current Streak**: 11 consecutive NO trades

### Code Path Issue Discovered

**The system is NOT running LLM FreeForm (Gen2) as expected:**
- Config: `use_gen2_engine: false`, `use_new_strategy: true`
- Active path: Legacy `new_strategy_trading.rs`
- Expected path: `gen2_engine.rs` with LLM FreeForm

### Database Issue

Hedge positions (opposite side of primary) were being created by legacy code:
```
Block 1773552600:
  - primary: YES trade (the actual decision)
  - hedge: NO position (risk management, now disabled)
```

Querying ALL positions included hedges, inverting the statistics.

### Corrected Recommendations

1. **Fix NO Trade Bias** - System is over-trading NO
2. **Enable Gen2 Engine** - If LLM FreeForm is desired
3. **Query Fix**: Always filter `WHERE position_id LIKE '%primary%'`
4. **Require 2x evidence for NO vs YES** (opposite of initial rec)

## Stats Reset Implemented (01:38)

### What Was Done

Added stats reset functionality for the Telegram `/status` command:

**Created:**
- `reset_stats.py` - Python utility to reset win/loss stats
- `stats_reset` table in database to track reset timestamp

**Modified:**
- `db_pool.rs` - Stats queries now filter by reset timestamp
- `telegram.rs` - `/status` command shows reset info

### How It Works

1. **Reset marker stored** in `stats_reset` table with timestamp and reason
2. **Stats queries** only count trades after the reset timestamp
3. **Telegram /status** shows:
   - Current balance and positions
   - Win/loss stats (since reset)
